systemd credentials are named, binary-safe key/value pairs passed to services at activation time. They replace the common pattern of putting secrets in environment variables or world-readable configuration files.

<CardGroup cols={3}>
  <Card title="Scoped access" icon="lock">
    Each credential is visible only to the service it was assigned to. The kernel enforces this on every access — there is no global secret store to compromise.
  </Card>
  <Card title="Non-swappable memory" icon="memory">
    Credentials are placed in `ramfs` when possible, so their contents are never written to disk as swap.
  </Card>
  <Card title="TPM2-backed encryption" icon="shield">
    Credentials can be encrypted with a key derived from the local TPM2 chip, ensuring they cannot be decrypted on any other machine.
  </Card>
</CardGroup>

## How services access credentials

When a service starts with one or more credentials configured, systemd sets the `$CREDENTIALS_DIRECTORY` environment variable to an absolute path where each credential is available as a plain file:

```
/run/credentials/myservice.service/
├── db-password
├── api-key
└── tls-cert
```

Service code reads credentials by opening files relative to `$CREDENTIALS_DIRECTORY`:

```sh
#!/bin/sh
DB_PASSWORD=$(cat "$CREDENTIALS_DIRECTORY/db-password")
```

The `%d` specifier in unit file settings expands to the same directory, so you can pass credential paths as environment variables or command-line arguments without hardcoding the runtime path:

```ini
[Service]
ExecStart=/usr/bin/myservice --cert=%d/tls-cert
Environment=DB_PASSWORD_FILE=%d/db-password
```

<Warning>
  Do not hardcode `/run/credentials/<unit-name>/` in service code. That path is only valid for system services. Use `$CREDENTIALS_DIRECTORY` instead so the code also works for user services.
</Warning>

## Providing credentials to a service

### LoadCredential=

Load a credential from a file on disk:

```ini
[Service]
LoadCredential=db-password:/etc/myservice/db-password.txt
```

The format is `<credential-name>:<source-path>`. At service activation, the file is read and its contents are made available under the given credential name.

You can also load from the system credential store by specifying a relative path — systemd searches `/etc/credstore/`, `/run/credstore/`, and `/usr/lib/credstore/`:

```ini
LoadCredential=db-password:db-password
```

### SetCredential=

Embed a credential value directly in the unit file:

```ini
[Service]
SetCredential=greeting:Hello, World!
```

<Warning>
  Unit files are world-readable on disk and over D-Bus. Use `SetCredential=` only for non-sensitive values such as public keys or configuration flags. For secrets, use `SetCredentialEncrypted=` instead.
</Warning>

### ImportCredential=

Import credentials from the system credential store without specifying a source path explicitly. Glob patterns are supported:

```ini
[Service]
ImportCredential=myservice.*
```

This imports all system credentials whose names match the pattern, making them available under the same names.

### LoadCredentialEncrypted=

Load an encrypted credential from disk and decrypt it transparently before passing it to the service:

```ini
[Service]
LoadCredentialEncrypted=db-password:/etc/credstore.encrypted/db-password.cred
```

The service receives the plaintext value. Decryption happens at activation time using the TPM2 chip and/or the host key stored in `/var/lib/systemd/credential.secret`.

### SetCredentialEncrypted=

Embed an encrypted credential directly in the unit file. Safe for sensitive values because the ciphertext cannot be decoded without access to the TPM2 and/or the host key:

```ini
[Service]
ExecStart=/usr/bin/systemd-creds cat db-password
SetCredentialEncrypted=db-password: \
        k6iUCUh0RJCQyvL8k8q1UyAAAAABAAAADAAAABAAAAC1lFmbWAqWZ8dCCQkAAAAAgAAAA \
        AAAAAALACMA0AAAACAAAAAAfgAg9uNpGmj8LL2nHE0ixcycvM3XkpOCaf+9rwGscwmqRJ \
        ...
```

Use `systemd-creds encrypt -p` to generate this inline form (see [Encrypting credentials](#encrypting-credentials) below).

## Encrypting credentials

The `systemd-creds` tool handles encryption and decryption. The encrypted output uses AES256-GCM.

### Encrypt to a file

```bash
systemd-creds encrypt --name=db-password plaintext.txt /etc/credstore.encrypted/db-password.cred
```

The `--name=` flag embeds the credential name in the ciphertext. This prevents an encrypted credential from being renamed and reused for a different purpose.

Reference the file in a unit with `LoadCredentialEncrypted=`:

```ini
LoadCredentialEncrypted=db-password:/etc/credstore.encrypted/db-password.cred
```

### Encrypt for inline use

```bash
systemd-creds encrypt -p --name=db-password plaintext.txt -
```

The `-p` flag formats the output as a `SetCredentialEncrypted=` line ready to paste into a unit file. The `-` argument writes to stdout.

### Decrypt

```bash
systemd-creds decrypt /etc/credstore.encrypted/db-password.cred -
```

Writes the plaintext to stdout. Only works on the machine where the credential was encrypted (or any machine with access to the same TPM2 state and host key).

## TPM2-based encryption

By default, when a TPM2 chip is available and `/var/` lives on persistent storage, `systemd-creds encrypt` uses a combination of two keys:

<AccordionGroup>
  <Accordion title="TPM2 key">
    Derived from the local TPM2 chip. Never stored on disk. Decryption requires physical access to the original hardware. Useful for credentials that must not be portable.
  </Accordion>
  <Accordion title="Host key">
    Stored in `/var/lib/systemd/credential.secret`, readable only by root. Generated automatically on first use, or explicitly with `systemd-creds setup`.
  </Accordion>
  <Accordion title="Combined key (default)">
    Both keys are required for decryption. The credential cannot be decrypted without the original TPM2 chip **and** the OS installation. This is the default when both are available.
  </Accordion>
</AccordionGroup>

To force a specific key type with the `--with-key=` flag:

```bash
# TPM2 only — required for credentials accessed from the initrd
systemd-creds encrypt --with-key=tpm2 --name=my-cred input.txt output.cred

# Host key only — useful when no TPM2 is present
systemd-creds encrypt --with-key=host --name=my-cred input.txt output.cred
```

<Note>
  For credentials consumed by generators or initrd code, use `--with-key=tpm2` or `--with-key=auto-initrd`. The host key at `/var/lib/systemd/credential.secret` is typically not available that early in boot.
</Note>

## Listing credentials

From within a service context, list the credentials passed to the current service:

```bash
systemd-creds list
```

Each credential is listed with its name, size in bytes, and security state (`secure` = ramfs-backed, `weak` = other memory, `insecure` = accessible by non-owner).

To view the credentials passed to the system as a whole (useful for debugging boot-time credential injection):

```bash
systemd-creds --system list
```

Display the contents of a specific credential:

```bash
systemd-creds cat db-password
```

Run a transient service with a credential and inspect it immediately:

```bash
systemd-run -P --wait -p LoadCredential=test:/etc/hosts systemd-creds cat test
```

## Complete unit file example

```ini
[Unit]
Description=My database-backed service

[Service]
ExecStart=/usr/bin/myservice

# Load a plaintext credential from disk
LoadCredential=db-password:/etc/myservice/db-password.txt

# Load an encrypted credential
LoadCredentialEncrypted=api-key:/etc/credstore.encrypted/myservice-api-key.cred

# Pass the credential directory path as an env var for a daemon that
# reads its password from a path in the environment
Environment=DB_PASSWORD_FILE=%d/db-password

# Recommended: enable mount namespacing so the credential directory
# is invisible to all other services
PrivateMounts=yes

[Install]
WantedBy=multi-user.target
```

## Reading credentials from service code

<Tabs>
  <Tab title="Shell">
```sh
#!/bin/sh

# Read a credential by name
DB_PASSWORD=$(cat "$CREDENTIALS_DIRECTORY/db-password")

# Or use the %d-expanded path from the environment variable set in the unit
DB_PASSWORD=$(cat "$DB_PASSWORD_FILE")
```
  </Tab>
  <Tab title="Python">
```python
import os

creds_dir = os.environ.get("CREDENTIALS_DIRECTORY")
if creds_dir:
    with open(os.path.join(creds_dir, "db-password")) as f:
        db_password = f.read().strip()
```
  </Tab>
  <Tab title="C">
```c
#include <stdlib.h>
#include <stdio.h>

const char *creds_dir = getenv("CREDENTIALS_DIRECTORY");
if (creds_dir) {
    char path[4096];
    snprintf(path, sizeof(path), "%s/db-password", creds_dir);
    /* open and read path */
}
```

## System credentials

Credentials can be passed to the entire system at boot rather than to individual services. These system credentials are then forwarded to individual services via `LoadCredential=` or `ImportCredential=`.

System credentials arrive through several mechanisms:

<AccordionGroup>
  <Accordion title="Container manager">
    A container manager sets `$CREDENTIALS_DIRECTORY` for systemd running as PID 1. `systemd-nspawn` supports this with `--set-credential=` and `--load-credential=`:

    ```bash
    systemd-nspawn --set-credential=passwd.hashed-password.root:$(mkpasswd mysecret) \
                   --set-credential=firstboot.locale:C.UTF-8 \
                   -i myimage.raw -b
    ```
  </Accordion>
  <Accordion title="Virtual machine (SMBIOS)">
    QEMU can pass credentials via SMBIOS OEM strings:

    ```bash
    qemu-system-x86_64 ... \
      -smbios type=11,value=io.systemd.credential:mycred=supersecret
    ```

    Binary credentials use the `.binary` variant with Base64-encoded values:

    ```bash
    -smbios type=11,value=io.systemd.credential.binary:mycred=c3VwZXJzZWNyZXQ=
    ```
  </Accordion>
  <Accordion title="Kernel command line">
    Pass credentials via the kernel command line with `systemd.set_credential=`:

    ```
    systemd.set_credential=mycred:supersecret
    ```

    **Avoid for sensitive data.** The kernel command line is readable by all userspace processes via `/proc/cmdline`.
  </Accordion>
  <Accordion title="UEFI / systemd-stub">
    Encrypted credentials placed on the EFI System Partition are picked up by `systemd-stub` and passed into the kernel. This enables secure parameterization of vendor-signed initrds.
  </Accordion>
  <Accordion title="initrd transition">
    Files placed in `/run/credentials/@initrd/` by initrd code are imported into the system credential set during the initrd → host transition.
  </Accordion>
</AccordionGroup>

View system credentials from a running system:

```bash
systemd-creds --system list
systemd-creds --system cat mycred
```

Propagate a system credential into a specific service:

```bash
systemd-run -p ImportCredential=mycred -P --wait systemd-creds cat mycred
```

## Credential search paths

When a relative source path is given to `LoadCredential=`, `LoadCredentialEncrypted=`, or `ImportCredential=`, systemd searches the following directories in order:

| Directory | Contents |
|---|---|
| System credential store (runtime) | Credentials passed at boot |
| `/etc/credstore/` | Administrator-managed plaintext credentials |
| `/run/credstore/` | Runtime plaintext credentials |
| `/usr/lib/credstore/` | Vendor-supplied plaintext credentials |
| `/etc/credstore.encrypted/` | Administrator-managed encrypted credentials |
| `/run/credstore.encrypted/` | Runtime encrypted credentials |
| `/usr/lib/credstore.encrypted/` | Vendor-supplied encrypted credentials |
